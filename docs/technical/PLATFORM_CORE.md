# Ядро и платформа прошивки Nixie Clock

## Цель документа
Этот файл описывает «ядро» прошивки, общую платформенную архитектуру и принцип работы с разными типами часов.

## Задачи ядра
Ядро должно обеспечить:
- независимость логики времени от типа вывода
- единую систему управления параметрами и меню
- поддержку разных аппаратных возможностей через профиль платформы
- устойчивую работу при отсутствии отдельных модулей (звук, дисплей, управление)

## Концепция платформенного профиля

Платформа определяется набором возможностей в `src/platform_profile.cpp`.
Это позволяет адаптировать ядро к разным устройствам без изменения базовой логики.

### Основные элементы профиля

- `display_enabled` — дисплей поддерживается
- `display_supports_nixie_tuning` — Nixie-дисплей поддерживает тонкую настройку
- `sound_enabled` — включен звуковой модуль
- `alarm_enabled` — включена подсистема будильников
- `controls_enabled` — доступно управление (кнопка/энкодер)
- `button_enabled` — доступна кнопка
- `encoder_enabled` — доступен энкодер
- `display_navigation_enabled` — поддерживается навигация по экранам дисплея

### Источники возможностей

Флаги берутся из двух мест:
1. статические аппаратные возможности устройства
2. пользовательские настройки из `config` (например `audio_module_enabled`, `ui_control_mode`)

### Пример адаптации

Если устройство не поддерживает аудио, то `sound_enabled` и `alarm_enabled` отключаются и соответствующие пункты меню не появляются.
Если тип часов не Nixie, то `display_supports_nixie_tuning` выключается.

## Компоненты ядра

### `platform_profile.cpp`

Файл реализует:
- `platformRefreshCapabilities()` — пересчитывает актуальный набор возможностей
- `platformGetCapabilities()` — возвращает текущую структуру возможностей
- `platformClockTypeName()` — отображение типа часов
- `platformUiControlModeName()` — отображение режима управления

### `config.h` / `config.cpp`

Конфигурация хранит параметры устройства и платформы:
- тип часов `ClockType`
- число разрядов `clock_digits`
- режим управления `UiControlMode`
- флаг `audio_module_enabled`

Эти параметры используют как меню, так и подсистема отображения/звук.

### `main.cpp`

Главная точка входа:
- инициализирует конфигурацию
- вызывает `platformRefreshCapabilities()`
- стартует основные менеджеры и задачи
- объединяет время, меню, дисплей, OTA и что ещё доступно

## Архитектурный принцип

Ядро следует принципу:
- `feature flag` — логика включена или выключена на уровне платформы
- `modular interface` — модули взаимодействуют через функции, а не напрямую через конкретный компонент
- `fallback` — при отсутствии функции приложение продолжает работать 

## Пример работы с возможностями

В `src/menu/menu_manager.cpp` проверяется, поддерживается ли аудиофункция:
- если `platformGetCapabilities().alarm_enabled == false`, меню будильников недоступно

В `src/alarm_handler.cpp` и `src/audio_task.cpp` проверяется:
- `platformGetCapabilities().alarm_enabled`
- `platformGetCapabilities().sound_enabled`

В `src/display/display_manager.cpp` может использоваться:
- `display_supports_nixie_tuning`
- `display_navigation_enabled`

## Типы устройств

В `include/config.h` перечислены поддерживаемые типы часов:
- `CLOCK_TYPE_NIXIE`
- `CLOCK_TYPE_NIXIE_HAND`
- `CLOCK_TYPE_CYCLOTRON`
- `CLOCK_TYPE_VERTICAL`
- `CLOCK_TYPE_MECH_2`
- `CLOCK_TYPE_MECH_PEND`

Это даёт основу для добавления новых аппаратных вариантов.

## Сценарии использования

### 1. Обычные Nixie-часы
- дисплей включён
- звук и будильники включены
- управление доступно кнопкой или энкодером

### 2. Часы без аудио
- `sound_enabled = false`
- пункты меню будильников и звука скрыты
- ядро продолжает работать как часы с дисплеем

### 3. Дисплей другого типа
- `display_supports_nixie_tuning` может быть выключен
- логика показов времени остаётся общей

## Точка расширения

Чтобы поддержать новый тип устройства:
1. добавить `ClockType` в `include/config.h`
2. настроить профиль в `platform_profile.cpp`
3. при необходимости добавить проверки в `platformRefreshCapabilities()`
4. адаптировать меню и подсистемы под новые возможности

## Рекомендации для документации `.docx`

- выделить раздел "Ядро платформы" как независимую архитектурную тему
- создать диаграмму зависимости `config` → `platform_profile` → `меню/будильник/дисплей`
- указать список флагов и их влияние на поведение
